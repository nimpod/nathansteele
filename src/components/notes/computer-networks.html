 1.2) Circuit-Switch/Packet-Switch:
  • The purpose of a computer network is to provide services to users.
  • Service may need...
    • guaranteed bandwidth --> circuit-switch
    • expected bandwidth --> packet-switch

  • Circuit-Switching:
    • Dedicated ptp between 2 devices.
    • Used in telephone systems.
    • You have to setup a path before you can start talking.
  • Packet-Switching:
    • Allows many devices to use the same link.
    • Before the message is sent, the message is broken down into chunks (aka. packets).
    • Used in the Internet.
    • There are 2 services in Packet Switching:
      • Virtual circuit - setup a circuit. Sender and receiver communicate directly (connection orientated).
      • Datagram - don't setup a circuit. Messages are sent immediately (connectionless).


       Connection-Orientated vs Connectionless Services:
    • Connection-orientated services:
      • Route chosen when virtual circuit is set up; all packets follow it
      • Packets arrive in order.
      • Each packet contains a short vc_id instead of its destination address.
    • Connectionless services:
      • Each packet is routed independently of its predecessors. Successive packets may follow different routes.
      • Packets may arrive in a different order from that in which they were sent
      • Each datagram must contain the full destination address.

    • Why? Because the listening range of a node is not large enough to listen to other nodes ether's, and therefore the CSMA algorithm breaks down in the case of a MANET.

 1.10) OSI model:
  • OSI = Open Systems Interconnection
  • The OSI model is a conceptual model that characterises and standardises the communication functions of a telecommunication or computing system.
  • It conceptually divides computer network architecture into 7 layers.
    • The higher levels cover network requests/responses, and representing data in meaningful ways.
    • Lower layers deal with electrical signals, binary data and routing data across networks.

  • 7) APPLICATION LAYER -  some application designed to be used by an end-user (e.g: SSH, FTP, SMTP, HTTP(S), DNS)
  • 6) PRESENTATION LAYER - this is the layer your OS works on. Provides common data abstractions to allow communication between two computers. Also does data encryption, converting to ASCII. (Note: this layer is not used now)
  • 5) SESSION LAYER - a conversation between two computers. Performs dialogue control, token management, data synchronisation. (Note: this layer is not used now.)
  • 4) TRANSPORT LAYER - decides how much data will be passed through. Encapsulates packets into segments. This is about security and validity. Scans the data to check for errors. Capable of error recovery, flow control, and re-transmission. Data that is not recovered from an error will not pass through, and may be destroyed.
  • 3) NETWORK LAYER - encapsulates the data into separate 'packets'. Also determines the best, most reliable, fastest route to deliver some data. (Routers are involved in this layer). The network layer also adds a source and destination IP address to each packet.
  • 2) DATA LINK LAYER - switches operate on this layer. Packets are turned into 'frames', and error-free data transfer of the frames is provided, over into the Physical Layer. MAC addresses!
  • 1) PHYSICAL LAYER - responsible for the transmission of digital data bits. Uses physical medium like copper cable, fibre optic cable, WiFi, Ethernet or infrared.

                  
  • Note: as you travel down the OSI model, you add headers to each layer:
  • Examples:
    • If your computer is not connected to the network => Application Layer problem
    • If a cable gets cut, or is mis-wired => Physical Layer problem
    • If you put in the wrong IP address => Network Layer problem
    • If the router dies => Network Layer problem

 2.2) Time Domain and Frequency Domain:

    • The domain (aka. x-axis) was measured in terms of time
    • The range (aka. y-axis) was measured in terms of amplitude

    • The domain (aka. x-axis) is measured in terms of each sine waves frequency.
    • The range (aka. y-axis) is measured in term of each sine waves amplitude

  • However, there is a maximum capacity that we cannot exceed.
  • A fancy jargon word for that maximum capacity is bandwidth.
    • Bandwidth = Max data rate of a channel (aka. carrying capacity)
    • Measured in BPS (bits per second)
    • The wider the bandwidth, the better. It means you can pick up a wider range of signals.

  • For example, suppose the bandwidth = 7
    • Now everything between 0 - 7 is received.
    • But anything after 7 will be discarded.

  • CONCLUSION:
    • We can never generate a perfect square wave in reality because of bandwidth limitations in the physical world. We can only generate approximations.
    • Narrow bandwidth --> bad (less data fits)
    • Wide bandwidth --> good (more data fits)

 2.3) Nyquists Theorem:
  • Nyquist's Theorem digitizes analog signals.
  • The therem does not work on noisy channels. In other words, it works on noiseless channels only!
    • "An analog signal waveform can be converted into digital by sampling the analog signal at equal time intervals."
    • "The sampling rate must be 'equal to, or greater than, twice the highest frequency component in the analog signal.”
    • https://blog.telegeography.com/what-is-the-nyquist-theorem-and-why-does-it-matter
  • Basically it says that the maximum data rate (i.e. bit rate) of a noiseless channel is 2x the bandwidth(Hz)
max data rate of channel = 2B
    Where B = bandwidth
  • Examples:
    • When B = 4MHz, ∴ I can push 8MB of data
    • When B = 8MHz, ∴ I can push 16MB of data
  • This basically implies that you can send 2 bits at a time instead of 1 bit at a time...
  • There is a slight change to the formula because of this:
max data rate of channel = 2Blog2V
    Where V = number of voltage levels
  • Examples:
    • When V = 4, 2Blog4 = 4B
    • When V = 32, 2Blog32 = 10B

Example using Nyquists Formula:
    • Engineers sometimes want to build a network, and they already know the bandwidth and max data rate of their channel. But they need to find out the value of V...
    • This can be calculated by simply using the 2nd formula, and rearranging it to find V.
    • In this example, bandwidth = 2, and max data rate = 16
B = 2
max data rate of channel = 16

∴ 16 = 2*4logV
∴ 16 = 8logV
∴ 2 = logV
∴ V = 4

  • The formula is:
max data rate of channel = Blog2(1+S/N)

    • A communications channel can be defined as a pathway over which information can be transmitted.
    • A channel could be a physical wire that connects communicating devices, radio, laser, or other radiated energy source that has no obvious physical presence.
    • All communication channels have a direction associated with them. There are 3 possibilities:
      1. Simplex
        • One way communication.
      2. Half-Duplex
        • Two way communication.
        • Only 1 system can talk at a time (e.g. walkie talkie). So sending and receiving cannot happen simultaneously. They must happen separately.
        • Essentially half-duplex is a simplex channel whose transmission direction can be switched.
      3. Full-Duplex
        • Multiple way communication.
        • In other words, you can transmit data in both directions simultaneously (e.g. telephone)
        • So you can send and receive at the same time.
        • Most modern communications use full-duplex transmissions.

     3.4.3) Parallel vs Serial Transmission:
    • Parallel Transmission:
      • Transmit a group of bits simultaneously.
      • Thick cables -> expensive over long distances.
      • Over long distances there can be synchronisation problems due to bit skew.

    • Serial Transmission:
      • Transmit 1 bit at a time.
      • You would expect this to be slower and less efficient than parallel transmission, but serial transmission is actually faster!
      • Serial transmission is more reliable and cheaper than parallel transmission, especially over long distances.
      • Unfortunately, serial transmission does add additional complexity...
        • The sender must determine the order in which the bits were originally sent.
        • And then the receiver must determine in which order to reconstruct a byte.
          • This like the difference between Big Endian (left->right), and Little Endian (right->left)
        • Basically, if the two devices do not agree on the same way to read the data (big endian or little endian) then things will go horribly wrong!

     3.4.4) Multiplexing:
    • If you want to send more than 1 conversation(s) across a single link, multiplexing is your friend.
      • Conversation = communication between 2 devices
    • Multiplexing allows the carrying of multiple signals on a single medium.
    • This allows more efficient use of the transmission medium, because more devices can use the link at the same time!
    • The multiplexer (MUX) on the left propagates the 3 conversations down the link.
    • The de-multiplexer (DEMUX) on the right separates the 3 conversations out again, and restore the original data.

    • There are 3 main multiplexing techniques:
      • Frequency Division Multiplexing (FDM) - share the channel by placing users on different frequencies.
      • Time Division Multiplexing (TDM) - share a channel over time.
      • Code Division Multiple Access (CDMA) - share a channel by giving out access codes.

          Frequency Division Multiplexing (FDM):
    • This is a multiplexing technique that shares the channel by simply placing users on different frequencies.
      • "break down the channel into multiple channels, each with a specific frequency, and we divide the channels among different hosts."
    • Divide by the frequency.
    • The total bandwidth available is divided into a series of non-overlapping frequency bands, each of which is used to carry a separate signal. Kind of like a motorway, where you can send multiple cars down one lane, and there can be multiple lanes per motorway.
    • For FDM to work properly frequency overlap must be avoided. Therefore, the link must have sufficient bandwidth to be able to carry the wide range of frequencies required.
                        
      • You can visualise each of those channels as the lanes of a motorway.
      • Then each signal that passes through those channels are like the cars travelling down the motorway.
                                (Explanation of motorway analogy =>  Wed Feb 13 10:35:00, 49:35)

          Time Division Multiplexing (TDM):
    • Use the entire bandwidth, but only for short time periods.
    • Instead of dividing the frequencies, we divide the time into separate slots. Then we give those slots to different conversations.
      • It's sorta like how Round Robin works.
      • They are taking turns in using the link. (i.e. each channel has a time slot for transmission on the link)
    • Takes advantage of the fact that achievable bit rate of medium exceeds the required data rate of a digital signal.

          Code Division Multiple Access (CDMA):
    • Shares the channel by giving user specific codes.
    • Distinguishing between conversations is just a case of differentiating between those access codes. You could then tune the network to only listen to conversations with a particular code.
    • Allows conversations to use the whole bandwidth at any time.
    • Commonly used in 3G networks.

4) DATA-LINK LAYER:
    4.1) Introduction to the Data-Link Layer:
    • Layer 2  - The Data-Link Layer to detect damage or loss of frames and recover from the problem
    • Those packets we talked about in the Physical layer are of no interest to us anymore. Instead they are grouped into frames. The frame is simply the transmitted element, in the form of a block of bytes.
    • More formally...
      • "The transmitted element in the Data-Link Layer is called a frame, which is regarded a block of bytes".
    • The sender breaks incoming packets from the Network layer into frames (packets ---> frames)
    • The receiver, using the frame will find out the first and last byte of the frame
    • The 3 main responsibilities of the data-link layer are:
      1. Framing - delimiting data units flowing on the channel
      2. Error control - detecting and correcting errors introduced from the physical layer (see section 4.2 on why frames go wrong)
      3. Flow control - prevents a fast sender from out-pacing a slow receiver

    • The 3 main features of the data-link layer are:
      1. To detect damage/loss of frames and recover
      2. To never let a damaged frame pass through the network

    • The 2 main services of the data-link layer are:
      • connection-orientated services
      • connectionless-orientated services

    4.2) When frames go wrong:
    • Frames can get lost easily. Here are four ways in which they can get lost:
      1. Short periods of disconnection (disconnection by a few milliseconds is enough to loose the entire frame.)
      2. Address of frame is damaged
      3. Software bugs
      4. Computer crash
    • Lot's of things can go wrong when sending frames and acknowledgements over a network. Here are 6 sequential scenarios in the form of questions:

    4.10) Flow Control:
    • Flow control is done in the Data-Link layer and the Transport layer.
    • What is flow control?
      • "It is the management of data flow between between nodes in a network so that the data can be handled at an efficient pace."
    • In Computer Networks flow control is applied by refusing additional node connections until the flow of traffic has subsided. There are numerous network protocols that are designed in order to achieve this refusal of nodes.
    • One of those protocols is the ARQ (Automatic Repeat reQuest) protocol. It is also known as the PAR (Positive Acknowledgement Retransmission) protocol.

        4.10.1) ARQ protocol:
    • The ARQ protocol works under 2 main assumptions:
      • Single-bit sequences are used (1001101110010001001)
      • There is an endless supply of data available.
    • Here is the psuedocode to describe the process of the ARQ protocol:

---------------------
sendFrame(A,B)                        // flow control
if (timeout) -> sendFrame(A,B)        // error control
reject(duplicate frame, duplicate ACK, frames with CRC errors)

      • TT = Transmission Time (time taken for frame to physically appear on the transmission link)
      • RTT = Round Trip Time (time taken to travel from A -> B -> A)

    • The best utilisation gives you a number that indicates how much percentage was used...
    • Disadvantages of ARQ protocol:
      • Requires 2 dataframes. One for the data, and one for the ACK.
      • You can only send 1 frame at a time
                             
        4.10.2) Piggy-Backed ARQ protocol:
    • The Piggy-Backing approach makes a simple change - combine the data & ACK into 1 dataframe.
    • This is why we call this the Piggy-Backed ARQ protocol, because the only difference is that ACK is literally getting a piggy-back on the data.
    • Advantages of Piggy-Backed ARQ protocol:
      • Fewer dataframes going forward and backwards --> better channel utilisation
      • Less computation
    • Disadvantages of Piggy-Backed ARQ protocol:
      • You can only send 1 frame at a time.
      • if (B has no data to send back to A) --> A will timeout because it is waiting for an ACK --> A will retransmit message --> poor utilisation
      • Solution to this problem: give both A and B a timer to ensure A does not wait forever.

      • sizeof(receiver window) = 1

    • Therefore the sender could send 8 frames before blocking, or 4 frames before blocking, or 18 frames before blocking, etc...
      • In general, this protocol allows the sender to send N frames before blocking.
      • The receiver only accepts frames that arrive in order.
      • But the receiver cannot accept frames out of order, or more than 1 frame at a time.

    • The utilisation for the Go Back N protocol is a massive improvement from previous protocols we have looked at.
    • The only difference with Go Back N, is the multiplication by N... The larger N is, the better the utilisation.
                    
    • When an error occurs in this protocol, the receiver discards all the subsequent frames coming after the erroneous frame.
    • The sender continues transmitting the frames unaware that the frames are being discarded by the data-link layer until timeout.
    • Then the sender will resend all the discarded frames.
                    
    • IMPORTANT:
      • If the window size = N,
      • there must be atleast n+1 sequence numbers.


        4.10.6) Selective Repeat protocol:
    • Improvement on the Go Back N protocol.
      • sizeof(sender window) > 1
      • sizeof(receiver window) > 1
    • In Selective Repeat you don't wait for an ACK before sending the next frame.
    • In other words, you can send multiple frames at once!
    • This protocol allows the sender to send N frames before blocking. (exactly the same as Go Back N)
    • The main difference between Selective Repeat and Go Back N, is that Selective Repeat doesn't discard the frames after the erroneous frame!
    • The subsequent frames are instead buffered by the data-link layer.

    • When an error occurs in this protocol, the receiver transmits a NAK to sender.
    • Then the receiver buffers the subsequent frames.
    • The sender continues transmitting the frames unaware until the NAK is received.
    • Sender re-sends frame 2.

      • BEB: used to decide how long to wait until next retransmission

    • Network links can be p2p or broadcast networks.

    • Channel Allocation Problem:

      • The solution(s):

        • Dynamic Channel Allocation - if (traffic is busy and unpredictable) --> use multiple access protocols.

        5.3.1) ALOHA:
    • ALOHA is a Random Access Protocol used in satellite networks.
    • It is the one case where broadcast networks are used in WAN's.
    • ALOHA protocols were developed in Hawaii, 1971. People on separate islands wanted to communicate with each other, but wired connections were too expensive.
    • They designed a ground-based radio system that use the ALOHA protocols.
    • There are two types of ALOHA protocols.
                    Pure ALOHA:
      • Users transmit whenever the data is ready to go. 
      • Users cannot sense the network.
      • It just sends when it wants to.
        • if (the frame is destroyed) --> sender waits a random amount of time and sends again
      • Disadvantages of Pure ALOHA:
        • Channels are used very poorly with this protocol, because all frames have the same size in ALOHA protocols.
        • Collisions happen when other users transmit during a vulnerable period that is twice the frame time (2T).
        • We call this the 'Critical Time'. So for ALOHA the critical time is twice the time we need too send a frame.
        • Solution:
          • no user should start sending during the T second period.

      • max throughput utilisation = 18%
        • bad under a High load -> lots of collisions
        • good under a Low load -> almost no collisions

                    Slotted ALOHA:
      • Utilisation is improved with Slotted ALOHA.
      • Still, users transmit whenever the data is ready to go.
      • Users still cannot sense the network.
      • So why is Slotted ALOHA better? It makes 2 changes:
        • Divide the time into slots. Each slot corresponds to 1 frame.
        • Users start sending at the beginning of their own time-slot.

      • max throughput utilisation = 37%
        • bad under a High load -> lots of collisions
        • good under a Low load -> almost no collisions

        5.3.2) CSMA:
    • There is 1 big disadvantage to both ALOHA protocols:
      • Users cannot sense the network...
    • Thing is, it is very difficult to listen to a satellite network before sending, because of the enormous distances signals must travel...

    • enter ... the ... CSMA = Carrier Sense Multiple Access protocol
    • This is a random access protocols that is used in LAN's (remember when I said CSMA was used in Ethernet cables! I was telling the truth!), and allows users to listen before sending.
    • This is my definition of the CSMA protocol:
      • "It verifies the absence of other network traffic before transmitting data on a shared medium. This prevents bit collisions from occurring."
    • CSMA is an improvement from ALOHA because users listen to the network before sending with CSMA.
    • Some people refer to CSMA as a polite protocol, because it listens before it talks. Very polite!
      • The 'Critical Time' for CSMA is the same as ALOHA - twice the time we need too send a frame.
        • In fact, for all collision detection methods, this will always be the answer!
        • Preventing collisions in the first place is a totally different story, and that would not be 2T (but that's kinda irrelevant in the current topic of conversation

    • There are 4 types of CSMA:

                    1-persistent:
      • This CSMA variant has a probability of 1 to transmit the complete frame when it can.
      • As soon as the idle channel has been detected, boom, transmission is done.
      • Here is the algorithm in pseudo-code:
if (there is data to send)
    listen to channel
    if (channel is busy) -> wait until idle
    if (idle channel is detected) -> transmit the complete frame     // transmit with a probability of 1, hence the name
    if (frame is damaged) -> wait random amount of time and start again
      • The algorithm is greedy, it sends everything as soon as idle.
      • Disadvantage of 1-persistent:
        • If 2 users are waiting, both will detect that it has become idle and then transmit at the same time -> bad utilisation under high load where potentially hundreds of users are waiting.

      • Max throughput utilisation = 55%
        • bad under a High load -> lots of users will wait
        • good under a Low load -> fewer collisions

                    Non-persistent:
      • This CSMA variant is less greedy than 1-persistent.
      • Users are very patient in Non-persistent. No two users will be allowed to talk at the same time.
      • Here's the algorithm in pseudo-code:
if (there is data to send)
    listen to channel
    if (channel is busy) -> wait random amount of time. Once that time is over, go to beginning again.
      • The algorithm provides fantastic utilisation, which results in fewer collisions.
      • Disadvantage of Non-persistent:
        • There will naturally be long delays when dealing with random waiting time.

      • Max throughput utilisation = 90%
        • good under a High load -> fewer collisions
        • bad under a Low load -> long delays results in slow network


                    P-persistent:
      • This CSMA variant is...
      • Applies only to 'slotted' channels (channels that are divided into slots)
        • 1 slot = 1 frame transmission time
      • Here's the pseudo-code:
if (there is data to send)
    listen to channel
    if (channel is idle) -> transmit with probability P        // so it will either do the transmission, or defer the transmission
      • Probability that the transmission occurs = P
      • Probability that it defers transmission = 1-P
        • For example:
          • P = 0.25, number_of_channels = 100,
          • Therefore, there is a 25% chance of each user transmitting immediately.
          • Thus, there is a 75% chance of each user deferring the transmission.

      • Max throughput utilisation = 90%
        • (depending on P) it is good under a High load 
        • (depending on P) it is good under a Low load

      • It is the most sophisticated variation of the CSMA protocol we have looked at so far.
      • Widely used on LAN's especially in classic Ethernet.

                    Binary Exponential Backoff Protocol:
      • This is the other limited contention protocol.
      • In addition to grouping up stations, we also group up time into slots (i.e. time division multiplexing).
      • if (collision occurs) -> time is divided into slots whose length = worst case round-trip propagation time on LAN cable.
      • So we keep dividing the time slots to reduce the probability of collision (2 stations sending at same time)

        • After 1st collision: each station waits either 0 or 1 time slot before retransmission.
        • After 2nd collision: each station waits either 0,1,2 or 3 time slots before retransmission
        • After 3rd collision: each station waits either 0,1,2,3,4,5,6, or 7 time slots before retransmission.
                                               // notice the pattern here? In general, the number of slots to wait is somewhere the interval [ 0 to (2^n) -1 ]
        • After 10th collision, we cannot go any further and the number is frozen at 2^10 -1
        • After 16th collision, report failure to computer. Further recovery handled by higher layers.

        5.3.5) Wireless LAN protocols:
    • Like with CSMA/CD, wireless LAN protocols don't need to deal with ACK's or NAK's.
    • Wireless LAN's have a short radio range, meaning a station cannot necessarily listen to every station in the network. Because of this, ACK's or NAK's are never really needed. 
    • Before, with wired LAN's there was a larger radio range, meaning ACK's and NAK's were needed.
    • Carrier sensing is still possible in wireless LAN's, but kinda useless due to something called the Hidden Terminal Problem

                    Hidden Terminal Problem:
      • Sender terminal(A) wants to send some data to receiver terminal(C)
      • Suppose A and C are outside each others range. An intermediate terminal(B) is required in order to transmit A's signal to C.
      • This means A and C are hidden terminals when transmitting to B.
      • When there are hidden terminals, there is the possibility of collisions occurring, because terminals will transmit data without realising that another terminal also transmitted data towards the same terminal.
      • This is an important problem in wireless LAN's.
                            

                    Exposed Terminal Problem:
      • Sender terminal(B) wants to send some data to receiver terminal(A)
      • Also, sender terminal(C) wants to send some data to receiver terminal(D)
      • Although it's not terribly clear on the diagram, terminal B is within terminal C's range.
      • This is bad news because C will falsely conclude that it cannot send its data to D, because it detected noise coming from terminal B.
      • This means B and C are exposed terminals when transmitting to A and D respectively.
                              

                    MACA protocol:
      • MACA = Multiple Access with Collision Avoidance
      • The MACA protocol is the solution to the Hidden Terminal Problem.
      • MACA (apparently) doesn't solve the Exposed Terminal Problem
      • Unfortunately collisions can still occur with MACA, but it mitigates collision issues a lot.

      • The whole reasoning behind the MACA protocol is summarized in the following sentence:
        • "Don't detect collisions. Avoid collisions."
      • When a station wants to start talking to another station, they must send send an RTS (Request To Send)
      • If the other station wants to say 'yeah sure lets talk' they must send a CTS (Clear To Send)
      • Now the two stations are able to freely communicate with each other.

        5.4.2) Transmission Time >= Round Trip Time:

    • Switch = bridges with more than 2 ports
    • Too build a larger Ethernet network, we must use Ethernet bridges...
    • Bridges read and learn MAC addresses of the devices in the network.
    • Bridges will reduce the size of each LAN segment, and hence the RTT.

    • Here's an example of an Ethernet network that uses bridges to connect 4 local segments:

    • The hexagonal nodes labelled B are the bridges.
    • The backbone is a fibre optic cable connecting the bridges together.
      • Suppose the orange device wants to talk to the blue device...
      • Orange will broadcast a frame to all devices in its local segment.
        • The 1st bridge will read the destination MAC address, and thus know that the blue is not in the 1st bridges local segment..
        • Send frame along the backbone...
        • The 2nd bridge will read the destination MAC address, and thus know that the blue is not in the 2nd bridges local segment..
        • Send frame along the backbone...
        • The 3rd bridge will read the destination MAC address, and thus know that the blue IS IN THE 3rd bridges local segment!
          • The 3rd bridge will broadcast the frame into this local segment, and then the blue device receives the frame.
    • So if any device (the blue device or the orange device for example) sends a frame, it is received by all devices and the bridge in the local segment we are currently in.
    • How is this all possible?
      • Because Ethernet bridges learn and relearn MAC addresses using a 'Forwading Table'. We call this ability Backward Learning.
      • Forwading tables keeps track of 2 things:
        • MAC addresses - the MAC address for each device in the network
        • Port number - every bridge has ports on it. Each device in a network connects to a port on the bridge.
      • There is 1 entry per network address.

6) NETWORK LAYER:
    6.1) Introduction to the Network Layer:
    • Layer 3  - The Network Layer is responsible for packet forwarding including routing through intermediate routers. It connects different networks built from different technologies using routers.
    • This layer encapsulates the frames from the Data Link layer into separate 'packets'.
    • The Network layer determines the best, most reliable, fastest route to deliver packets through routers around the world.
    • The network layer also adds a source and destination IP address to each packet.
    • Protocols used in the Network layer:
      • IP
      • DNS
      • ARP / Bootp / DHCP

    • Unfortunately, we cannot build a world wide network using switches and MAC addresses only...
    • Why? Switches have their disadvantages:
      • A sender and receiver cannot talk to each other if they sit in different networks using different technologies.
        • (aka: switches don't work across more than 1 link layer technology)
      • Switches don't scale to large networks
      • Switches don't perform traffic control (they might choose a bad route that is very congested. They don't analyse traffic)
    • Because of these limitations of switches, we have the Network Layer...

    • The Network Layer performs internetworking: 
      • When many different networks built from different technologies are combined into a single network.
      • This forms a network of networks (i.e. The Internet)

    • Routers do 2 things:
      • routing - process of discovering network paths. Filling in the table... ( global )
      • forwarding - sending of packets along a path. Looking into the table... ( local )

    6.2) Services:
    • The Data-Link layer is the layer below the Network Layer.
    • Data-Link layer provides connectionless services to the Network Layer.
    • Specifically the Data-Link layer provides the Datagram service to the Network Layer.

        6.2.1) Connectionless Service - Datagram:
    • Important things:
      • Packets are routed independently. So each packet may end up going on a very different journey to another packet...
      • With datagrams, you don't need to setup a path.
      • The packet contains the full source and destination address (where did the packet come from, and where is it going?!)

    • Here's an example of how the datagram service works and how routing tables work:
                            
    • The original message is broken down into 4 packets, labelled { 1, 2, 3, 4 } on the diagram
    • Packets { 2, 3, 4 } travel on the same path (A->C->E->F)
    • But packet { 1 } went on a different path (A->B->D->F). Why? Possibly due to congestion or network failures.

    • So let's suppose P1 wants to send a message to P2
      • Being humans, it's pretty obvious how to get there, just traverse through the nodes in an order which yields the lowest total weight.
      • Unfortunately, for computers it doesn't see the graph like us humans. Hence the need for routing tables.
    • Reading the routing tables is pretty easy:
      • Start at table A: to get to router F, we must go through router C...
      • Continue from table C: to get to router F, we must go through router E...
      • Continue from table E: to get to router F, we must go through router F...
      • And we have arrived at router F!
    • Once the message arrives at the destination host (P2), we must read the MAC address of the host device.
    • We can read a MAC address using the Address Resolution Protocol (ARP), which simply maps IP addresses to MAC addresses.

        6.2.2) Connection-Orientated Service - Virtual circuit:
    • Important things:
      • Packets typically follow the same path.
      • With virtual circuits, you must setup a path.
      • Like with datagrams, the virtual circuit is split into many packets. The packet contains a vc_id.
      • Multi-Protocol Label Switching (MPLS) protocol makes virtual circuits.

    • Here's an example of how virtual circuits work:
                                    
    • P1 and P3 both decide to send data to P2.
    • Initially, both P1 and P3 have vc_id = 1
    • This is because they do not know that each other exist yet, so having the same vc_id is no problem.
    • When they come onto the same path at router A, the vc_id for one of the virtual circuits must change.
    • We do this by just changing P3's vc_id = 2
    • Then P1 and P3's virtual circuits continue on the same path towards P2.
    • Basically...
      • if two routers are suddenly on the same path, change the vc_id of one of them so that you can distinguish between them.
    • Reading the tables is pretty intuitive:
      • in - what was the previous router, and what was the vc_id for this virtual circuit?
      • out - what is the next router, and what was the vc_id for this virtual circuit?

    6.5) Packet Fragmentation:
    • In the Internet routers break down packets into smaller fragments - aka. Packet Fragmentation
    • There are 2 types of packet fragmentation:
      • Transparent Fragmentation - routers reassembled at boundary
      • Non-transparent Fragmentation - packet is only reassembled at destination.

            6.5.1) Transparent Fragmentation:
      • Routers reassemble packet at boundary.
      • Receiver has no idea where the packet was changed, because it only gets the final version and is told nothing about how it was reassembled.
      • There are 3 problems with transparent-fragmentation:
        • Computational work - each router does a lot of work - breaks packet into fragments, reassembles fragments into segments.
        • Time delays - Router must receive all fragments from the broken down packet before it can continue.
        • Fragments are repeatedly reassembled many times during transmission - router must receive all fragments from the broken down packet before it can continue.
      • Basically the transparent approach is shit because it is a brute force version of packet fragmentation...
      • Due to these disadvantages we do not use it in The Internet.

            6.5.2) Non-transparent Fragmentation:
      • Packets are only reassembled once at the receiver/end destination.
      • This gives a more optimal solution compared to the transparent approach, since now we don't have to keep waiting at the router for all the fragments to appear!
      • There is 1 tiny-weeny problem with non-transparent fragmentation...
        • If one fragment is lost, all fragments will be resent from the beginning again.
      • It's less of a problem really, but more of a solution to a potentially bad situation.

            6.5.2) No-fragmentation: path MTU discovery:
      • Sooo apparently packet-fragmentation isn't actually used.
      • Great. Well I hope I haven't just wasted the last 2 hours of my life making those awesome tables.
      • Researchers argued that fragmentation is too slow and tedious (I agree!!!) and that can be detrimental to performance because of:
        • header overheads
        • a whole packet is lost if any of its fragments are lost
      • We can avoid doing fragmentation all together by using Path MTU discovery.
      • Path MTU Discovery basically determines the max transmission unit(MTU) size on the network path between two IP hosts, usually with the goal of avoiding IP fragmentation.
      • They are built using the correct size in the first place, meaning we don't need to do fragmentation anymore.
      • Doing path MTU discovery may cause an extra delay at the beginning, because of all these calculations we have to do.
      • So basically, path MTU discovery allows us to calculate and use an optimal size.

      • Here are the steps involved in Path MTU discovery, for IPv4:
1) Each IP packet is sent with its header bits set to indicate that no fragmentation is allowed. (and set DF = 1).
    2) If a router receives a packet that is too large, it generates an error packet and returns it to the source, and drops the packet.
    3) When the source receives the error packet, it uses the information inside to re-fragment the packet into pieces that are small enough for the router to handle.
4) Repeat steps 2 and 3 until the MTU becomes small enough to traverse the entire path without fragmentation.
5) We get an optimal MTU! Yay.

    6.7) Internet Protocol (IP):
    • IP is the common language over the Internet. Kinda like how English is the common language in Europe.
    • The IP protocol was designed in the late 1970's by Vint Cerf and Bob Kahn.
    • The Internet is a large collection of different kinds of subnet's. IP is like the 'glue' that allows all these subnet's to communicate and exchange data.
    • IP provides delivery of packets from one host to another host in The Internet, even if the hosts are in different networks.

    • An IP Address is a unique identifier for any device connected to a computer network such as the Internet.
    • Currently there are two versions of IP addresses in use today:
      • IP Version 4 (IPv4)
      • IP Version 6 (IPv6)

            6.5.1) IPv4:
      • IPv4 has been the common IP version of the Internet since 1983 January 1st.
      • There are 4,294,967,296 (approx. 4.3 billion) unique IP addresses with IPv4.

      • IPv4 addresses are 32-bits or 4-bytes long.
      • They are displayed in a dotted-decimal format that contains four octets, with each number in range 0-255
        • Example = 54.225.229.200
        • Lowest = 0.0.0.0
        • Highest = 255.255.255.255
      • Each IPv4 address consists of (netID, hostID)
        • Example = 54.225.229.200
        • Lowest = 0.0.0.0
        • Highest = 255.255.255.255
      • The netID identifies a network.
      • The hostID identifies the specific host on that network.

      • All IP hosts on the same IP network must have the same netID
        • This reduces the size of the routing table.
        • Why? Because there is 1 entry per network.
        • If there was 1 entry per host, that would be a bigggg table!

            IPv4 format:
    • There are lots of part that go into the IP address structure for IPv4...
                 
    • The upper section where most of the fields are is the header:
      • Version = 4
      • IHL = stands for Internet Header Length, calculated by len(header)
      • Differentiated services = specifies type service supplied to this packet (e.g. real-time)
      • Total length = len(datagram)
      • Fragment Offset = if you don't want routers to fragment your packet, set fragment offset = 1
      • MF = 1 if the fragment is the final fragment, else 0
      • DF = starting bit-position relative to payload
      • Identification = unique ID to distinguish IP packets including all their fragments.
      • Header Checksum = a CRC checksum for the header only. Recalculated at each router.
      • Protocol = which transport layer protocol should be used for this packet (UDP? TCP?)
      • Time to live = how long does the packet live for? Prevents a packet from roaming around the Internet forever causing congestion. Each time this packet reaches a router, decrement this number by 1. if time_to_live = 0 then drop the packet.
      • Source address = 32-bit IP address of sender.
      • Destination address = 32-bit IP address of receiver.
      • Options = used to extend the functionality of IP (not used much anymore...)
    • The lower section sitting on its own is the payload:
      • payload = data from the transport layer

      • Used to route a packet to the correct LAN.
      • Once you arrive at the LAN, you need to know the MAC address of the address.
      • So in a nutshell, ARP maps an IP address to a MAC address.

      • The ARP protocol is only used by the border routers in a network.

      • We don't know the IP address though, we only know its domain name...
      • How do we convert domain names -> IP addresses?
        • DNS server!
      • So host 1 will contact the DNS server for the CS network and asks for the IP address of eagle.cs.uni.edu
      • The DNS server replies with an IP address of 192.32.65.4, which as you can see on the diagram corresponds to host 2.

      • Now we need that MAC address...
      • Host 1 knows that 192.32.65.5 is on local ethernet (the CS network).
      • ARP
      • Host 1 broadcasts message "are you 192.32.65.5?"
      • Host 2 replies with "yeah that is me, also my MAC address is E2"
      • /ARP
      • Host 1 puts IP packet inside an Ethernet frame....
        • Header of the frame = E2

            6.9.3) Reverse Address Resolution Protocol (RARP):
      • RARP assigns an IP address to the host.
      • It used to be done manually:
        • If you wanted to connect to a new network, you would talk the network administrator (admin) in person.
        • The admin has a table in the router that maps the users MAC address -> IP address
        • A RARP server is needed on each network because broadcast messages are not forwarded by routers.
      • Like I said though, it is manual.
      • It is no longer used.
      • What is the difference between ARP and RARP?
        • ARP retrieves the physical address of the receiver
        • RARP retrieves the IP address for a computer from the server.

            6.9.4) The Bootstrap Protocol (BOOTP):
      • Bootp does EXACTLY the same thing as RARP (assigns an IP address to the host)
      • HOWEVER, moving from one LAN to another LAN no longer requires manual input. Changing the IP address depending on your network is done automatically in Bootp.
      • There is no static database here.
      • When a Bootp server receives a request, it looks in its database for a matching entry and then returns the result to the host.
      • Also, Bootp uses the UDP transport layer protocol. UDP messages are passed by routers.
      • Disadvantages of Bootp:
        • You still have to manually create the table initially
        • Each time a new host is added to the LAN, we can't actually use Bootp until an admin has assigned it an IP address and manually entered its IP and MAC addresses into the tables.

            6.9.5) Dynamic Host Configuration Protocol (DHCP):
      • Oh apparently we don't use RARP or Bootp anymore. Sweet glad I've just spent like 3 hours trying to understand them.
      • DHCP does EXACTLY the same thing as RARP and Bootp (assigns an IP address to the host)
      • The main difference with DHCP is that you no longer have to manually enter into the table the IP address. This was the big disadvantage of RARP and Bootp.
      • This is how DHCP achieves this:
        • 1) Broadcast a 'discover packet' inside the LAN.
        • 2) A DHCP relay on the local LAN forwards the discover packet to a (possibly remote) DHCP server.
        • 3) The DHCP server replies with an IP address for the host.

    6.10) Routing Algorithms:
    • Routing algorithms are responsible for deciding which output line an incoming packet should be transmitted on.
    • In other words, what path is best suited for the packet?
    • The best path is dependent on many factors, which the algorithms must take into account when choosing the best path for a particular packet.
    • The algorithms will fill in routing tables to document the process.

    • There are 2 types of routing algorithms:
      • Adaptive algorithms - can change routing decisions depending on network topology and traffic (ie. dynamic)
      • Non-adaptive algorithms - makes a routing decision and does not change it (ie. static)

    • Routing algorithms are dependent on 4 main factors:
      1. Topology - packet must be sent in the right direction
      2. Speed of interconnections - avoid slow lines
      3. Apparent load of target hosts - avoid slow machines
      4. Congestion - avoid busy routes

    • All routing algorithms have the following 3 properties:
      1. Robustness - must be able to cope with network changes, network topology changes, hosts going down or fluctuating traffic levels.
      2. Stability - converge to equilibrium.
      3. Fairness - routers should have the same chance as getting traffic through them (ie: traffic should be equally split among routers).
      4. or Optimality - the route should transport a packet to the destination as quickly as possible.
        • Fairness and Optimality contradict each other ( hence the 'or' ). You can't have both at the same time.

    • Routing has 2 phases:
      • Routing - routers exchange information to decide on best routes, using routing tables.
      • Forwarding - when the packet arrives at the router we forward it onto another router (makes sense now!)

    • The routes that an algorithm computes could either be a datagram or a virtual circuit.
    • Recalling their properties...
      • if packets follow each other -> virtual circuit.
      • if packets are routed independently -> datagram.


            6.10.1) Flooding:
      • Flooding is similar to broadcasting. You can visualise it like 'flooding' a network with loads of packets.
      • The algorithm sends every packet out onto every outgoing line (like broadcasting).
      • It does not send the packet back in the direction it came from.

      • The main problem with flooding is that it generates lots of duplicate packets.
      • We have 3 strategies to control flooding:
        1. Hop Counter (aka. time to live):
          • <best solution>
          • Implement a hop counter in the packet header.
          • It is initialised to the number of intermediate nodes in the network.
          • It is decremented at each hop.
          • When hop counter = 0, throw the packet out because it has been in the network for too long. Be gone demon-packet! Be gone!
        2. Use sequence numbers:
          • Watch each packet's sequence number and make sure that the same packet is not sent twice.
        3. Selective Flooding:
          • Broadcasting the packets to every router isn't necessary most of the time.
          • To improve, we only send the packets to routers that are approximately in the right direction.
          • Routers that are way behind or in a totally different direction to the current direction will not receive the packet.

      • Advantages of flooding:
        • very robust - if a large amount of routers were destroyed at once flooding could recover (useful in military applications for this reason!)
        • always chooses shortest path - 
        • good for small amounts of data - 
      • Disadvantage of flooding:
        • not practical in most applications -
        • very bad for large amounts of data - the network will collapse if you try to send a 2MB file across the network.

            6.10.2) The Optimality Principle:
      • The most optimal route through a network.
      • It is a general statement about optimal routes, ignoring topology and traffic.
                                
      • The optimal path to get from I to K is through I -> J -> B -> K
      • Since J and B are on the same optimal path, J to K and B to K would follow the same route.
      • So what this principle tells us is that a node can determine its best route by learning its neighbours best route.

            6.10.3) Link State Routing:

      • It uses Dijkstra's Algorithm because it knows the topology of the entire network.

        5. Compute the shortest path (min delay) to every other router.

            6.10.5) Distance Vector Routing:
      • Distance Vector Routing (DVR) is an Adaptive Routing Algorithm.
        • This means that DVR has no problems adding new routers to the network.
        • However this doesn't mean that DVR is good at removing routers.
      • DVR was used in the ARPANET, the predecessor to the Internet.
      • In the DVR algorithm, routers only know their neighbours. The routers do not necessarily understand the whole network at the beginning (ie. at round 1).

                    Count-To-Infinity (C2I):
      • A major concern that DVR has to deal with is the 'Count-to-infinity' (C2I) problem:
        • DVR reacts rapidly to good news (new better route)
        • DVR reacts slowly to bad news (best route goes down)
        • When a router goes down, the distance from all routers to it tends to infinity, increasing with each cycle.
        • It will take lots of cycles before a router is declared unreachable.
        • The amount of time it takes for a router to become unreachable is called the 'convergence time'.
                                        
        • With good news, the delay between a router and its neighbor is only 1 (ie: good news travels fast)
        • With bad news, the delay between a router and its neighbor propagates through the network and will diverge to infinity (ie: bad news travels slowly)

                            Solution to C2I problem:
        • How do we solve the issue of bad news diverging to infinity?
        • Set inf to a number...
          • inf = 'longest path' + 1
          • **also equivalent to**
          • inf = 'number of routers'
        • Whatever the number is, it will now represent the longest path in the network.
        • If a routers counter reaches this number, it means that this router is no longer reachable.
        • Therefore, every other router will delete the entry for this unreachable router.

        • Each router knows nothing about the internal structure of other regions.

            6.10.7) Internetwork Routing:
      • Just to recap what an Internetwork is... "An internetwork is a combination of multiple networks that use different technologies. They are connected using border routers"
      • To do routing in an internetwork, there are 2 levels to the routing algorithm:
        1. Within each network, an Interior Gateway Protocol (IGP) is used for routing.
        2. Across the networks that make up the Internet, and Exterior Gateway Protocol (EGP) is used.
          • The networks in an internetwork can use different IGP's
          • But the networks MUST use the same EGP.

            6.10.8) Routing in the Internet:
      • IGP:
        • A common IGP distance vector protocol is the Open Shortest Path First (OSPF).
        • OSPF uses an LSR algorithm, and falls into the IGP group of protocols.
        • OSPF replaces an old distance vector protocol called the Routing Information Protocol (RIP).
        • RIP was not a great protocol because:
          • it doesn't cope well with changes to network topology.
          • used the hop counter to calculate distance between 2 routers.
        • OSPF is a great protocol because:
          • dynamic to changes in network topology.
          • can handle real-time traffic.
          • does load balancing - splits load over multiple lines to prevent congestion.
          • supports hierarchical routing.
          • incorporates security measures - routers are vulnerable to attacks, but using a good protocol like OSPF you can mitigate security concerns.

      • EGP:
        • The main Internet EGP distance vector protocol we use is the Border Gateway Protocol (BGP).
        • Here are the main things that all EGP's do:
          • Politics - where in the world is the network traffic going? It may not allow local traffic to be routed through certain other networks or countries.
        • BGP provides another solution to the C2I problem:
          • Inform its neighbours about the paths it is using.
          • It does this by exchanging the path to the destination. Then the destination knows where to route it!

      • LSR uses flooding.

      • DVR does not use Dijkstra, because it does not know the entire topology beforehand. Only neighbors.
      • DVR uses routing tables that are updated by exchanging information with its neighbors.
      • DVR suffers from the Count-to-infinity problem.


    6.11) Congestion Control:
    • Congestion occurs when too many packets are present in the subnet.
    • If congestion is too high, routers begin loosing packets.
    • The routers loose packets because they are forced to throw away packets without inspecting them because there are no buffers left to store them.
    • Congestion may be caused by 2 things:
      1. Network topology
      2. Hardware

            6.11.1) Congestion that is caused by network topology:
      • Suppose we have a star topology network like the one shown below:
                                
      • In such a network, it might be hard to predict data rate coming towards the central router.
      • This is because there are 3 routers that could all potentially send data at the same time.
      • This means the buffer could fill up very quickly, meaning packets will got lost quicker, resulting in congestion.
      • Most people will say "why not use a bigger buffer?"
        • For a given router, if the time taken for a packet to travel from the back of the buffer to the front of the buffer is > timeout period, the router will resend the packet.
        • This results in duplicate packets.
        • Causing 'Congestion Collapse' (when a network cannot send or receive anymore packets)
        • So by using a bigger buffer, we managed to make congestion even worse!

      • Congestion tends to feed upon itself and become worse. Like a rotting corpse, you can't stop it from rotting when it's off.
      • Let's suppose we add 4 more routers to this network (Router 5, Router, 6, Router 7 and Router 8):
                                
      • Router 7 sends data to Router 6, which sends data to Router 5, which sends data to Router 4...
      • So if Router 4 becomes congested routers 5, 6 and 7 will also become congested.
        • Those red numbers indicate how many packets each router wants to send (e.g. Router 2 wants to send 14 packets)
        • Router 4 will send all 10 packets at once, and will then wait for an ACK (assuming Router 4 uses a sliding window protocol) 
        • Router 4 will save a copy of these 10 packets in its own buffer.
        • Buffer of Router 4 becomes congested.
        • Therefore, Routers 5, 6 and 7 become congested.
        • The congestion has spread through the network causing 'Backpressure' (when a router becomes congested, the congestion feeds upon itself and spreads in the network)

            6.11.2) Congestion that is caused by hardware:
      • Slow processors in routers can also cause congestion.
      • It is a common issue in networks that have machines of varying processing speeds (some slow, some fast, some normal)
      • Network engineers usually try to make the network speed < backbone speed
        • This is why the network uses twisted pair cables, and the backbone uses fibre optic cables.

            6.11.5) Open Loop vs. Closed loop:
      • Open Loop and Closed Loop are a collection of different solutions to control congestion.
      • OPEN LOOP solutions - wait for sender and receiver to sense that there is something wrong in the network. 

      • CLOSED LOOP solutions - ask the network itself for some feedback on its status.
        1. Monitor the network to detect when and where congestion occurs, such as:
          • Percentage of all packets discarded for lack of buffer space.
          • Average queue lengths.
          • Number of packets that time out and re-transmitted.
          • Average packet delay.
          • Standard deviation of packet delay.
        2. Distribute this information to places where action can be taken:
          • Bad solution - create a choke packet that travels from the congested router to the sender. This is a bad solution because sending another packet through the network will add even more congestion! 
          • Good solution - set the Explicit Congestion Notification (ECN) field in the IP header to 11.
        3. Adjust network operation to correct the problem:
          • there are 5 different approaches to adjusting the network due to congestion...

            6.11.6) Approaches to Congestion Control:
      • These are the 5 approaches to dealing with congestion control:
        1. Network provisioning - very preventative
        2. Traffic-aware routing - preventive
        3. Admission control - somewhere in the middle
        4. Traffic throttling - reactive
        5. Load shedding - very reactive
                            
      • They are usually applied on different time scales to either prevent congestion or react to it once occurred.

                    1) Network provisioning:
      • Is the network going to be good?
        • Is it well matched to the traffic that it carries?
        • Are the routers fast enough?
      • Turn on spare resources as needed.

                    2) Traffic-aware Routing:
      • If the previous approach failed...
      • Compute best route using Dijkstra's Algorithm, taking into account the traffic.
        • Shift traffic away from congestion hotspot's by changing the shortest path weights.
        • Split traffic across multiple paths if necessary.
      • This approach is called 'Traffic-aware' because of the fact that it considers the network traffic in Dijkstra's Algorithm.

                    3) Admission Control (traffic shaping):
      • If the previous approaches failed...
      • Allow new traffic to be loaded only if the network has sufficient capacity.
        • This reduces the overall traffic.
      • Make hosts transmit at a uniform rate (aka. do traffic shaping)
        • This is because one of the main causes of congestion is that traffic is often bursty. Hosts often don't transmit at consistent rates.
        • This approach is often called 'Traffic Shaping', and is considered part of this Admission Control approach.
        • There are 2 Traffic Shaping algorithms that I will cover shortly. These are the...
          • Leaky Bucket Algorithm
          • Token Bucket Algorithm

                    4) Traffic Throttling (choke packets):
      • If the previous approach failed...
      • Router creates a choke packet, and sends it back to the sender.
      • The choke packet is a way of telling all of the senders to 'SLOW DOWN!!! THERE'S TOO MUCH STUFF IN THE NETWORK'
      • After the sender receives the choke packet, the sender will reduce its TR in attempt to slow down.
      • This process may be repeated several times until the router is satisfied with the state of congestion in the network.
      • There are 2 ways of doing this:
        1. A choke packet that affects only the sender. Intermediate routers ignore the choke packet.
        2. A choke packet that affects each router it passes through. 

                    5) Load Shedding:
      • If the previous approach failed...
      • The last resort if all other approaches failed is to make routers start dropping packets.
      • How you drop packets depends on the application being developed.
      • There are policies you should follow that dictate which packets to drop, how to drop them and when to drop them.
      • There are 2 ways of doing this:
        1. Wine approach - drop the newest packet.  Used in applications that require reliability (e.g. email, viewing a webpage, downloading files)
        2. Milk approach - drop the oldest packet. Used in applications that require multimedia (e.g. video games, live-streams, video calling)


            6.11.7) Leaky Bucket Algorithm:
      • This is a Traffic Shaping algorithm.
      • Let's begin with an analogy, using a literal bucket:
                            
        • There is a tap which has water coming out of it.
        • The water falls into a bucket.
        • The bucket has a hole in it.
        • The water leaks out of the bucket at a constant rate, regardless of fast or slow water is going into the bucket.
        • That rate is dependent on the size of the hole.

      • Now let's replicate this analogy in the context of a network:
                                
        • The host (aka. sender) is equivalent to the tap.
        • The sender can send packets at any rate it likes.
          • In this diagram, it send 3 packets, then stops, then send 2 more packets.
          • This is labelled as 'Unregulated flow'.
        • The packets passes through the 'bucket'.
        • The packets leave the bucket at a constant rate.
          • You can see this constant rate after leaking out of the bucket.
          • This is labelled as 'Regulated flow'.
        • Depending on your application, you may or may not want a regulated flow.

            6.11.8) Token Bucket Algorithm:
      • This is another Traffic Shaping algorithm, an alternative to the Leaky Bucket Algorithm.
      • Allows saving up tokens to the maximum size of the bucket, N.
        • This means that bursts of up to N packets can be sent at once, allowing some burstiness in the output stream, and giving faster response to sudden bursts of input.
        • This is great, but means the Token algorithm is vulnerable to large bursts if N is too big.
        • A common solution to this it to use both the Token and Leaky algorithms (in that order).
          • Sooo in summary, to get the smoothest traffic possible, put a Leaky bucket after the Token bucket.
      • In Token Bucket, we are no longer restricted to a regulated flow of data.
      • Instead, the flow of data can speed up or slow down depending on the size of the burst.
      • It has the word 'token' in its name because each packet picks up a token on its way through the bucket. The packet can only progress into the network if it picked up one of these tokens.
                                
        • The host (aka. sender) sends packets at any rate it likes.
        • The packets passes through the 'bucket'
        • One token (those black things in the bucket) is added to each packet as it passes through.
        • According to [diagram a] we have 5 packets, and 3 tokens in the bucket to begin with.
        • Therefore only 3 of the packets will be given tokens. Those 3 packets will pass through into the networks.
        • The remaining 2 packets will have to wait behind the bucket for more tokens to be given out.

7) TRANSPORT LAYER:
    7.1) Introduction to the Transport Layer:
    • Layer 4  - The Transport Layer is responsible for delivering data with the desired reliability end-to-end between two application processes anywhere on the network. It has a similar role to the Data-Link layer - it's all about reliability.
    • This layer encapsulates the packets from the Network layer into separate 'segments'.
    • This layer cares about nothing else other than sending data from a sender to a receiver. That is all the transport layer cares about.
    • We call the Transport layer an 'end-to-end' layer because there are no intermediate devices that have any involvement in this layer.
      • Network layer had intermediate routers between sender and receiver.
      • Data-Link layer had intermediate switches between sender and receiver.
      • Physical layer had intermediate hubs between sender and receiver.
      • Because there are no intermediate devices it only needs point-to-point connections, hence end-to-end.
    • You can see this clearly on this snippet of the OSI model:
                    

    • For a reliable service we need acknowledgments, timeout and retransmission.
    • Protocols used on this layer are:
      • UDP
      • TCP
      • DCCP

    • port + IP = socket (a socket is a combination of port and IP address. It is gives a way of uniquely identifying each process in a machine)


    7.2) Segment:
    • A segment has two sections:
      • header - contains the port number of sender and destination.
      • payload - coming from the application layer.
    • In fact the packet and frame also have a header and payload.
    • The segment, packet and frame are all nested within each other in such a way that would be complex to explain but easy to visualise:
            
    • Look at it carefully:
      • The segment as a whole is nested inside the Packet payload
      • and the packet as a whole is nested inside the Frame payload
    • And just for your information:
      • Segment header - port number of router (for sender and dest)
      • Packet header - IP address of switch (for sender and dest)
      • Frame header - MAC address of hub (for sender and dest)

    7.3) Transport Layer vs Data-Link Layer:
    • I said earlier that "the transport layer has a similar role to the Data-Link layer - it's all about reliability"
    • Sooo why do we even have the transport layer?
    • It does exactly the same roles as the Data-Link layer (error control, flow control, reliability) so why the extra effort?
      • Reason 1) The layer below us right now, the Network layer is unreliable (because of the IP protocol). The users have no control over the subnet, so the only way to improve the QoS is to put another layer on top of the Network layer.
      • Reason 2) If a programmers want to write a network application, they need do work on a higher level than the network layer, because networks vary a lot and you don't want your application to only work for one particular type of network.

    • Main differences between Transport Layer and Data-Link Layer:
      • Data-Link Layer each outgoing line uniquely defines the destination host. But the Transport Layer must specify a full destination address.
      • Connection and disconnection is more complex at Transport Layer (3-way handshake) compared to Data-Link layer.
      • It is difficult to define an accurate timeout interval at the transport level due to queuing delay, different buffer sizes and different speeds. (This means using Sliding Window protocol is much harder at Transport Layer)


    7.4) Connection:
    • When a host wants to communicate to another host, it must set up a transport layer connection.
    • Setting up that connection can be done in 2 ways:
      • Bad - a transport entity in source host sends a connection setup segment to the destination which returns an acceptance segment. Bad because of unreliability of the Internet, causing duplicate/lost/corrupt packets.
      • Good - 3-way handshake with sequence numbers

    • How the bad solution fails:
      • CR = Connection Request
      • ACK = Accept Connection
                            
      • The CR is duplicated in the following diagram because it decided to take a very congested path.
      • Because of the delays due to congestion, A thinks the CR packet is lost, so A re-sends CR.
      • So now we have 2 CR's in the network.
      • Eventually both of them will arrive at B.
      • B thinks 2 connections have been requested, when in reality only 1 was requested. Hence duplicate CR!

    • How the good solution succeeds in providing a reliable connection:
      • Use large independent sequence numbers to uniquely identify each CR and ACK, preventing duplicates.
      • Also uses a 3-way handshake...

        7.4.1) 3-Way Handshake:
    • How does it work?
      • We are going to use the diagram below as an example.
      • It shows the normal operation of the 3-way handshake:
                                 
      • There are 2 hosts, Host 1 and Host 2.
      • These are the steps that happen...
        • 1) Host 1 sends a CR, which has a sequence number of x.
        • 2) Host 2 replies with an ACK of x to prove that it detected CR. The ACK has a sequence number of y.
        • 3) Host 1 replies with the actual data with a ACK of y to acknowledge the acknowledgement. The ACK has a sequence number of x. (the actual data is attached using piggy-backing)

      • For the future of this network, x will be the starting sequence number for data sent by Host 1,
      • and y will be the starting sequence number for data sent by Host 2

    • What problems does the 3-way handshake solve?
      • Duplicate CR - rejects old sequence numbers.
      • Duplicate Data - rejects data with the wrong ACK.

        7.4.3) Initial Connection Protocol (ICP):
    • Some service processes will always be listening for CR's on known addresses.
    • Other processes may not actually exists when a CR is made and will not be listening.
    • Makes sense right? CR gets lost because no one is dealing with it... Frekin' lazy processes.
    • Solution -> Initial Connection Protocol (ICP).
      • "ICP uses a process server which exists on machines that wish to offer services to remote users."
    • What is a process server? -> A process server basically just listens to lots of ports simultaneously.
    • When is it used? -> Every host that wants to offer services to other services in the Internet will have a process server.

    • Let's look at how ICP works with a diagram...
                    
    • This is how a user process in Host 1 establishes a connection with a mail server in Host 2 via a process server.
      • 1) Host 1 sends a CR to Host 2.

        7.4.4) Sockets:
    • Services are obtained by creating end-points, called sockets.
    • Sockets are a combination of an IP address and a port number:
      • IP: used by Network Layer to uniquely identify the host.
      • Port number: used by Transport Layer to uniquely identify the process on the host.

    • We have around 65,000 different port numbers.
    • Port numbers between 0 - 1023 are called 'well-known ports' also known as 'system ports'.
    • Well-known ports are used for popular protocols like SMTP, DNS, HTTP.
    • If you were developing some kind of a network application, you would use ports between 1024 - 6500 because those ports are not allocated to anything by default.

    7.5) Disconnection:
    • We have covered connection, so we know how to setup the initial connection and how to exchange data during its lifetime.
    • Now the 2 hosts must terminate the connection (i.e. disconnect).
    • There are two ways to approach disconnection:
      • Symmetric -
        • Both hosts keep separate connections to each other.
        • Meaning you could break the connection to one another easily.
        • No loss of data.
        • This is how TCP works.
      • Asymmetric - 
        • Either host may terminate the connection.
        • When one host breaks hangs up, the connection is broken.
        • Potential loss of data.
          • Solution -> 3-way handshake
        • Potential loss of Disconnection Requests (DR)
          • Solution ->
            • Use a timeout variable (terminate the connection if no packets have been sent for 'timeout' number of seconds) (for example: when Barclays mobile banking app times-out if you're inactive for 30 seconds)
            • Hosts transmit 'keep-alive' packets to keep a connection open when they have no data to send.
        • Like in telephone systems.
                            


    7.6) Multiplexing:
    • Multiplexing is all about the sharing of data...

        7.6.1) Upward Multiplexing:
    • Allow multiple applications to use the same IP address.
    • You don't want to be limited to just 1 application to be connected to the Internet at one time. Most users these days have hundreds of applications that are competing to use the Internet.
    • The Transmission Control Protocol (TCP) does Upward Multiplexing for us.
                
      • As you can see from the diagram, we do indeed have one IP address at the Network Layer.
      • The multiple applications that want to use the Internet sit in the Transport Layer.
      • When a packet arrives to the IP address at the Network Layer, the packet is not broadcasted to each app.
      • Instead, depending on which port number this packet is specified to, it will decide which app to send it to.
        • The port number is found in the payload of the packet.

        7.6.2) Downward Multiplexing:
    • Basically the reverse of Upward Multiplexing.
    • There is 1 application in the Transport Layer, and it wants to spread its data out to several different IP addresses across the Internet.
    • The Stream Control Transmission Protocol (SCTP) does Upward Multiplexing for us.
                
      • The connections are made using virtual circuits.


    7.7) UDP:
    • User Datagram Protocol (UDP)
      • Connectionless.
      • Unreliable.
      • Message stream.
                    
    • Used for sending and receiving data...
    • More specifically, it is used in real-time multimedia applications, such as Skype, Messenger video calling, Twitch, BT Sport football live-streams, Internet Radio, or when you want to receive a request from a DNS server.
    • UDP was designed for "one request, one response" applications.
    • It's very fast because of this philosophy. We don't want to spend lots of valuable time checking acknowledgments, re-transmitting data, doing error control or flow control.
    • This is why it is popular for real-time applications!
    • Also, if you think about all the times when you have been watching online live-streams and the video skips and stops working constantly, that is UDP in action! The unreliability of the protocol shines brightly in these sorts of situations where it is constantly stopping, starting and skipping and can't quite seem to settle down and behave like a normal stream. UDP is sending data, but it doesn't really care what happens to it. UDP simply cares about speed and transmitting the data quickly.

    • UDP's high speed is due to a combination of many things that I have already mentioned:
      • Connectionless - it doesn't waste time setting up a connection
      • No acknowledgements - 
      • No sequence numbers - 
      • No error or flow control -
      • No re-transmissions -
      • No checksum - the checksum in the header is rarely used as it requires a lot of computation.

        7.7.1) UDP Header:

      • As you can see, we are doing nothing to do with sequence numbers, acknowledgements, error or flow control.

    • Jitter is when a real-time service such as a the video and audio from a stream becomes all jumpy, glitchy, keeps stopping and starting, and generally being annoying.

      • Instead, we stream packets at the 'right time'.

      • High jitter -

        • Therefore it is hard predict the buffer.
        • This may result in missing packets, because the range doesn't always accommodate for all packets.
      • Low jitter -
        • Gaps between packets are small.
        • Therefore it is easier to predict the buffer.

            7.9.4) TCP Congestion Control:
      • The Network layer is responsible for congestion control, specifically by TCP.
      • (high capacity receiver) + (slow network) -> use Congestion Control.

      • Congestion can be caused by:
        • Small capacity receiver.
        • Internal congestion inside the network. (ie: a busy network).

      • The general approach to preventing congestion is -
        • Don't inject data into the network if it is likely to be dropped.
        • Introduce a 2nd window, called the congestion window (maintained by the sender):
          • To determine how many bytes it may send, the sender takes the minimum of the receiver window and the congestion window.
            • The receiver fills in its own receiver window.
            • The sender has to fill in the congestion window.
            • There are many algorithms that do this for us.
            • We will study the Slow-Start and Adaptive Increase Congestion Control algorithms:

            7.9.5) Slow-Start Congestion Control Algorithm:
      • The sizeof(window) rises exponentially.

      • sizeof(window) = maxSegmentSize.
      • If an acknowledgement is received, we double the sizeof(window).
      • We keep doing this, and eventually packets will be lost, indicating congestion.
      • if a timeout occurs -> reset sizeof(window) = maxSegmentSize
                                
      • This algorithm is inefficient, and the transmission rate is very low.

            7.9.6) Adaptive Increase Congestion Control Algorithm:
      • An optimised version of Slow-Start, where after a certain point the sizeof(window) rises linearly.

      • sizeof(window) = maxSegmentSize.
      • Adaptive Increase begins its algorithm using Slow-Start.
      • When we hit a certain window size referred to as the 'threshold', the rate of increase becomes linear.
      • We keep doing this, and eventually packets will be lost, indicating congestion.
      • if a timeout occurs ->
        • threshold = 1/2 * (sizeof(window))
        • sizeof(window) = maxSegmentSize
                                
      • In this diagram, a packet is lost when the sizeof(window) = 40
      • Therefore, the threshold is (1/2 * 40) = 20

      • The Adaptive Increase algorithm is an improvement on Slow-Start.
      • It's still not perfect... but who cares these are the only 2 algorithms we will be focusing on here.

    • It verifies the absence of other network traffic before transmitting data on a shared medium. This prevents bit collisions from occurring.
    • Here is the pseudo-code:
listen to 'ether'            // notice it is called the ether, hence the name ethernet
if (noisy)
    --> wait and start again at 1
else if (quiet)
    --> simultaneously transmit a packet of data and listen to 'ether'
    if (what you hear is not what you transmitted)
        --> wait and start again at 1
else
    --> successful
    • So if the network is noisy, then the sender will wait, and will try again.
    • Eventually when the network is quiet, the sender will transmit the data but keep listening to the ether to prevent bit collisions.

What does it mean?
Backpressure
Backpressure refers to the buildup of data at an I/O switch when buffers are full and not able to receive additional data.
It is a resistance or force opposing the desired flow of data through software.

Sample answer...
If a router has no free buffers, it must ignore newly arriving packets. When a packet is discarded, the sending router times out and retransmits, until the packets get through. This router must then keep the packets longer in its queues, so it becomes itself congested and the problem spreads (called backpressure).
Adaptive Routing algorithms
Adaptive algorithms attempt to change routing decisions to adjust to traffic changes or topology changes.
(e.g. Distance Routing Vector algorithm, Dijkstra's Algorithm, etc...)
Upward Multiplexing
If only one network address is available on a host, all transport layer connections have to use it (Transport connections share a network address).
Downward Multiplexing
Multiple network addresses are available to 1 host application. The host application can spread its data to each network address. (Network connections share a host application). This type of multiplexing is Implemented using the SCTP protocol.
Jitter
When an unreliable network is supplying a real-time service, and packets are streamed by their arrival time, the video stream will jitter.
Jitter means the video and audio could be jumpy, stopping and starting, and displaying packets out of order.
We use buffers to reduce jitter in real-time transport protocols.
Border router
A router than connects two networks together. It sits on the border of a network, holding onto a connection with another network.
Internetworking
This is when you combine different networks that use different technologies. They are connected using border routers.
Congestion Collapse
When too large of a buffer is used, if the time taken for a packet to travel from the beginning -> end of the buffer is > timeout period (which is more likely with a larger buffer), packets will be lost.
Piggy backing
Piggy backing is when you combine the data and acknowledgement into one dataframe. The ACK literally gets 'piggy-backed' on the data.
Means there are less dataframes going backwards and forwards, resulting in a better channel utilisation. Also more optimal computation.
It is used to improve the ARQ protocol.

Time to live (TTL) field
One of the fields in the IPv4 header. It determines the amount of time that a packet should remain in a network for.
TTL is a number that is decremented by 1 each time it arrives at a new router.
When TTL = 0, the packet is discarded forever.
The TTL field reduces congestion because packets will not roam around the network forever.
Choke packet
Used in the congestion control approach called 'Traffic Throttling'.
If a router thinks the network is too congested, the router will send a choke packet to the original sender. 
The choke packet is a way of telling the sender to lower its transmission rate.
This process may be repeated several times, until the router is satisfied with the state of congestion in the network.
Home agent vs. Foreign agent
The Home agent keeps track of the devices that are leaving the network.
The Foreign agent keeps track of the devices coming into its network.

Binary Exponential Backoff protocol

My original answer
This is a limited contention protocol
It groups up station into groups and time into slots (Time Division Multiplexing)
If a collision occurs, the time is divided into slots whose length is the worst case round trip time on a LAN cable.
We keep dividing the time slots to reduce the probability of two stations colliding (transmitting at same time)

  • After the 1st collision, each station waits either 0 or 1 time slots before retransmission.
  • After the 2nd collision, each station waits either 0,1,2 or 3 time slots before retransmission
  • After the 3rd collision, each station waits either 0,1,2,3,4,5,6,7, or 8 time slots before retransmission.
  • After the 10th collision, the number is frozen at 2^10 -1(1023)
  • After the 16th collision, report failure to computer.
Sample answer
After the first collision, each host independently waits either 0 or 1 slot times before trying again. If two stations collide and each one picks the same random number (0 or 1), they will collide again.  

After the second collision, each one picks either 0, 1, 2 or 3 at random and waits that number of slot times.

If a third collision occurs, then the number of slots to wait is chosen from the interval 0 to 2^3 – 1.   

After 10 collisions, the number is frozen at 2^10 - 1(1023) slots.

After 16 collisions, report failure to the computer. Further recovery handled by higher layers.
